IFC・データ連携

Tekla／Solibri IFC → GH インポータ群

IFC を Grasshopper の中へドサッと取り込む——メッシュ＋属性つきの DataTree で、マルチスレッドだから速い。

稼働中 GH Python3 ノード ＋ Hops サーバー 要 ifcopenshell

MOTION PREVIEW

動作デモ

PREVIEW⟲ LOOP 10s

IFC を全コアで並列に形状変換 → メッシュを DataTree に整列 → 各パーツへ属性を添付して Grasshopper へ、という流れを10秒でループ再生しています。

1なにをするツール？

Tekla や Solibri が書き出した IFC ファイルを、Grasshopper の中へメッシュとして取り込むインポータの集まりです。取り込むのは形状（メッシュ）だけでなく、各部材の属性（GlobalId・Name・IfcClass・Pset・数量など）も keys / values の DataTree として一緒に持ち込みます。

ポイントは マルチスレッドで速いこと。IFC の形状生成は重い処理ですが、ifcopenshell で全コアを使って一気に変換します。さらに、GH キャンバスをフリーズさせずにフル CPython（GH の外）で処理する Hops サーバー版も用意されていて、巨大な IFC でも Grasshopper が固まりません。

Tekla の IFC は「アセンブリ（部材のまとまり）→ パーツ」という階層を持つので、{アセンブリ; パーツ} の入れ子 DataTree に整えます。階層を持たない標準的な IFC 用には、IfcClass（IfcBeam / IfcColumn…）単位でまとめる Solibri 版もあります。

※ DataTree＝Grasshopper のデータを枝分かれで管理する構造。／ Hops＝GH の計算を外部の Python サーバーに肩代わりさせる仕組み。／ Elefront＝メッシュに UserText（属性）を焼き付けてベイクする定番プラグイン。

3使い方

  1. IFC を用意するSolibri／SMC 抽出ツール で取り出した IFC など、取り込みたいファイルのパスを用意する。
  2. 直取込ルート（手軽）GH に tekla_ifc_importer（標準 IFC なら solibri 版）を置き、ifc_file_path を渡して solve。メッシュと属性の DataTree が出る。
  3. 重い IFC は Hops ルートキャンバスを固めたくない・多コアで回したいときは py tekla_ifc_hops_server.py を起動し、GH の Hops コンポーネントの Path に http://localhost:5000/tekla_ifc を入力。
  4. 属性を分解するHops 経由なら返ってきた AttributesJSON を tekla_ifc_attr_parser に通して keys / values に分解する。
  5. 属性つきでベイクgeo_tree と keys/values を Elefront の SetUserText → Bake に渡し、UserText つきのメッシュとして Rhino へ焼く。
  6. 後工程へできたメッシュ＋属性を、部材検索やモデリング支援の各ツールへ流す。

4つながり（ワークフロー）

前工程 Solibri／SMC 抽出ツール群
↓ Tekla／Solibri IFC→GH インポータ群（このツール）

SMC から取り出した IFC を Grasshopper に持ち込む橋渡しです。ここで属性つきメッシュにしておくと、部材検索や図面化の各ツールがそのまま属性を使えます。

5実データでの手順

🧵 手元の Tekla IFC で試す

専用サンプルはありません。実案件の Tekla IFC（アセンブリ階層つき）で試すと、DataTree の枝分かれと属性の乗り方が一番わかりやすいです。

  1. 小さめの IFC を用意まずは部材数の少ない Tekla IFC で。軽量化ツールで形状だけにすると読み込みが速い（※ただし属性は消える点に注意）。
  2. 直取込で構造を見るtekla_ifc_importer を置いて solve。Panel で geo_tree の枝を開き、{アセンブリ; パーツ} になっているか確認する。
  3. Hops で速さを体感サーバーを起動して Hops 経由に切替。GH が固まらないまま多コアで変換される様子を比べる。

6注意点・既知の罠

  • ⚠ マルチスレッド × OpenCASCADE のクラッシュに注意

    並列で形状生成すると OpenCASCADE がクラッシュしやすいため、settings オブジェクトはスレッド内で毎回独立生成する設計になっています（コード内で対策済み）。改造するときはこの前提を崩さないでください。

  • ⚠ 巨大 IFC は Hops サーバー版で。GHPython 直は重い

    GHPython の直取込は、非常に大きい IFC だと GH ごと固まることがあります。要素数が多いモデルは フル CPython の Hops サーバー版を使うと、多コアで速く・キャンバスをフリーズさせずに処理できます。

  • ⚠ Hops は初回に pip インストールが必要

    Hops サーバーは pip install ghhops-server flask rhino3dm ifcopenshell が前提です。GH ノード側は先頭の #! python 3 と # r: ifcopenshell 宣言（RH8 Python3 ノード）が必要です。

  • ⚠ Tekla 版と標準 IFC 版を取り違えない

    Tekla 版は IfcElementAssembly 階層がある前提です。階層を持たない標準 IFC（意匠モデル等）に使うと枝が組めないので、その場合は solibri_ifc_importer（IfcClass 単位）を選んでください。
