1なにをするツール？

別のツール 「ライノセラスに変換」（Tekla の IFC を自動で仕分け・マスター更新するパイプライン）が、いま動いているのか・成功したのか・どこで止まったのかを、ブラウザ画面で見えるようにする監視ダッシュボードです。

仕組みはシンプル。変換パイプラインが処理を終えるたびに、裏で Webhook（8765番ポートへのお知らせ）を飛ばします。このツールがそれを受け取り、状態を信号のような色（緑＝完了／黄＝実行中／赤＝失敗）とイベント履歴で表示します。さらに死活監視（watchdog）で「サーバが落ちていないか」を見張り、異常時にはTeams通知を送る仕掛けも入っています（通知の宛先設定は現状プレースホルダのまま）。

※ Webhook＝プログラム同士が「これが起きたよ」と自動でお知らせし合う仕組み。／ 死活監視（watchdog）＝サーバやプロセスが生きているかを定期チェックすること。

3使い方

  1. サーバを起動するC:\Python1\BIM_Dashboard\webhook_server.py を実行。ローカルの localhost:8765 で待ち受け開始。
  2. ダッシュボードを開くブラウザで http://localhost:8765/ を開く。ここが状態表示の画面。
  3. 変換パイプラインを回す「ライノセラスに変換」 側を実行すると、処理のたびに notify_dashboard() が状態をPOSTしてくる。
  4. 状態を見る状態円が緑（完了）／黄（実行中）／赤（失敗）に切り替わり、イベント履歴が積み上がる。/api/status で機械的に取得も可能。
  5. 通知を有効化する（任意）Teams通知を使うなら teams_config.py の ALERT_WEBHOOK_URL / MEMBER_WEBHOOK_URL に実際の宛先を設定する（現状は空のプレースホルダ）。

4つながり（ワークフロー）

監視対象 IFC自動変換（ライノセラスに変換）
↓ BIM Dashboard（このツール）

「ライノセラスに変換」 が実行結果を送ってくる受け皿・見える化係です。送信元は from BIM_Dashboard.webhook_sender import notify_dashboard の形でこのフォルダをimportしているため、フォルダ名や場所を変えると送信側が壊れます（下の注意点を参照）。目的が近い BIM_Shield とは今のところコード連携はありません。

5注意点・既知の罠

  • ⚠ フォルダ名・場所を変えると送信側が壊れる

    別プロジェクト「ライノセラスに変換」が from BIM_Dashboard.webhook_sender import notify_dashboard, notify_progress とパッケージimportしています。C:\Python1\BIM_Dashboard\ を移動・改名すると、そちらのimportが即エラーになります。progress.json / events.json / pipeline_state.json も相対パスで初期化される対象なので消さないこと。

  • ⚠ Teams通知は未設定（そのままでは飛ばない）

    teams_config.py の ALERT_WEBHOOK_URL / MEMBER_WEBHOOK_URL は2件ともプレースホルダのままで、実際にTeamsへ通知された形跡はありません。通知を使うなら本物のIncoming Webhook URLを設定してください。

  • ⚠ サーバが起動していないと記録されない

    ダッシュボードのサーバが立っていないと、送信側は「サーバー未起動のため記録をスキップ」となり履歴に残りません。監視したいときは先に webhook_server.py を起動しておいてください。

  • ⚠ 現在ほぼ休止（3.5ヶ月ログなし）

    直近の稼働記録が 2026-03 で途切れています。壊れてはいませんが、実運用として使うかは要判断です。目的の近い BIM_Shield との将来的な統合候補という位置づけです。
