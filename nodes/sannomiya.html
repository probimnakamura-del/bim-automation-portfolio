メール要約転送システム sannomiya

特定案件のメールをAIが自動で要約して、添付ごと社内へ転送・記録してくれる常駐アシスタント。

稼働中（20分ごとに監視） スタンドアロン / Python Gmail × ローカルAI（Ollama）

MOTION PREVIEW

動作デモ

PREVIEW⟲ LOOP 10s

受信トレイの監視 → AIによる要約と添付解析 → 社内転送とログ記録、という一連の自動処理を10秒でループ再生しています。

1なにをするツール？

特定案件のメール対応を丸ごと自動化する常駐システムです。Gmailの受信トレイを20分ごとに監視し、対象ラベルの付いたメールが届くと、そのスレッドと添付ファイルをAIが要約・解析します。

要約ができたら、添付ファイルを Google Drive／Box Drive 経由でTrimble Connect の同期フォルダへ保存し、社内メンバーへ整形したメールを転送。あわせてスプレッドシートと Obsidian にログを残します。つまり「案件メールを見て・要約して・関係者に共有して・記録する」までを、人が張り付かなくても回してくれる仕組みです。

※ 要約AIは既定でローカルのOllama（qwen3-vl:8b-instruct）を使い、うまくいかないときだけGemini APIへ自動フォールバックします。／ Obsidian＝メモをMarkdownで貯めるノートアプリ。ここでは案件ログの保管先として使っています。

3使い方

  1. 認証情報を置くGoogle API の credentials.json と token.json を C:\Python1 直下 に置く（初回はブラウザ認証で token を発行）。
  2. 設定を確認するsannomiya\config.py で監視ラベル・転送先・AIバックエンド（既定 ollama）などを確認。ローカルAIを使うなら Ollama を起動しておく。
  3. 起動するランチャー sannomiya_pro_sync.py を実行（py sannomiya_pro_sync.py）。以降は20分ごとに自動巡回し続ける常駐モードに入る。
  4. まず空振りで試すいきなり本番転送が不安なときは sannomiya\dry_run.py を使う。副作用なしで取得・要約だけを確認できる。
  5. 結果を見る社内への転送メール・スプレッドシートの追記行・Obsidianのログで処理結果を確認。エラーは sannomiya\error.log に残る。

4つながり（ワークフロー）

メール要約転送システム（このツール）

このツールは単独で完結して動きます。他のGHノードとファイル入出力で連携するのではなく、Gmail・Google Drive・Sheets・Obsidian・Trimble Connect といった外部サービスと直接やりとりする自己完結型の常駐システムです。

5実際に動かすには

📧 専用サンプルモデルはありません（実データ＝Gmailで動きます）

3Dモデルを使うツールではないため .3dm のサンプルはありません。動作確認は本物のGmailに対して、まず副作用のないドライランで行うのが安全です。

  1. テスト用ラベルで試す本番の転送先を触らずに済むよう、config で監視ラベルをテスト用に切り替えるか、自分宛のメールで確認する。
  2. ドライランsannomiya\dry_run.py を実行。取得・要約までを副作用なしで確認できる。
  3. AIバックエンドを切り替えて比較要約品質を見たいときは config の AI_BACKEND を ollama / gemini で切り替えて出力を比べる。
  4. 本番監視へ問題なければ sannomiya_pro_sync.py を常駐起動し、20分ごとの自動巡回に任せる。

6注意点・既知の罠

  • ⚠ 認証情報は C:\Python1 直下から読む

    credentials.json と token.json は sannomiya フォルダの1つ上（C:\Python1 直下）から読み込まれます。フォルダ内に置いても認識されないので配置場所に注意してください。

  • ⚠ 本番転送は不可逆。まずドライランで確認

    いちど社内へ転送したメールは取り消せません。ラベルや転送先の設定を変えた直後は、必ず dry_run.py か自分宛メールで動作を確かめてから常駐させてください。

  • ⚠ Ollama が動いていないと Gemini に切り替わる

    既定はローカルの Ollama ですが、モデル未起動などで失敗すると自動で Gemini API（クラウド）へフォールバックします。社外にデータを出したくない場合は Ollama が確実に起動しているか確認を。GPU で速く動かすには CUDA 設定（4090環境の罠）も参照してください。

  • ⚠ error.log は自動で消えず溜まり続ける

    実行エラーは sannomiya\error.log に追記され続けます（ローテーションなし）。長期常駐で肥大化することがあるので、ときどき中身を確認・整理してください。

  • ⚠ 起動用の .bat はリポジトリ外にある可能性

    起動用の .bat（例：メール転送.bat）が C:\Python1 配下に見当たらないため、実運用の起動元は別の場所（タスクスケジューラ等）の可能性があります。常駐が動いていないときは起動元の在り処を確認してください。
